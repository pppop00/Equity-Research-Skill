# 中文研报风格指南

供 Phase 5 中文 HTML 研报撰写器（`agents/report_writer_cn.md`）使用。文风参照国内头部券商（中金公司、中信证券、海通证券、国泰君安）的深度研究报告规范。

---

## 文风与语气

**应做到：**
- 专业、客观、简洁——这是机构研究报告，不是科普读物
- 结论前置——先说核心判断，再展开支撑逻辑
- 有数据支撑的判断——每个结论都应有具体数字或事实依据
- 逻辑链完整——从数据到分析到结论，层次清晰

**应避免：**
- **把 Markdown 写进 HTML 占位符**：最终报告是静态 HTML，不会解析 `**加粗**`、`*斜体*`、`` `代码` ``；读者会看到未转换的星号与反引号。摘要、趋势卡、`{{LATEST_OPERATING_UPDATE_TEXT}}`、`{{GEO_REVENUE_TEXT}}`、投资逻辑与 Sankey 说明等字段一律用**纯中文书面语**；若必须强调，用「」括注或极少量 `<strong>…</strong>`（与模板协调）。
- 口语化表达：不用"其实"、"说白了"、"总的来说"等
- 模糊语言：不用"较大"、"显著"、"较为"等无法量化的修饰（除非后跟具体数字）
- 感叹号与情绪化表达
- 无依据的正面评价："领先行业"、"一流水平"需有数据支撑
- 机器翻译腔：不要逐字翻译英文，用中文金融业内的惯用表达

## 各章节写作规范

### 投资摘要

**四段通用长度：** 每段 **160–200 个汉字**（不含空格和标点可近似检查），纯中文书面语，不写 Markdown。四段都要有明确职能，避免互相重复。

**第一段（`{{SUMMARY_PARA_1}}`）：** 将原来的公司基本面介绍与最新财务表现合并：主营业务、核心收入引擎、收入规模、同比增速、利润率或现金流质量、上市地/板块（如适用）。这一段回答「这家公司是什么、刚交出了怎样的财务结果」。

**第二段（`{{SUMMARY_PARA_2}}`）：** 必须来自 **`edge_insights.json` → `summary_para_2_draft`**。用三步写法：表面读法 → 行业潜规则/特殊规则/口径重构 → 投资含义。读者读完应明确知道「这个数据为什么不能按普通口径理解」。禁止写成泛泛的行业趋势或公司优势。

**第三段（`{{SUMMARY_PARA_3}}`）：** 保持原有核心逻辑职能：主要催化剂、行业趋势、战略布局、模型或客户需求变化，并写清约束项（监管、供给、价格、客户预算、竞争等）。不要重复第二段的 edge insight。

**第四段（`{{SUMMARY_PARA_4}}`）：** 保持原有行业内定位与结构职能：**细分赛道/子行业边界**、**市场份额**（尽量给出可查的**多年序列**并标明统计口径与来源；若无第三方份额，诚实说明并改用公司分部收入或定性表述）、**主要运营/生产与研发布局**、**收入主要来源地**（须与 **`financial_data.json` 地区披露**一致，或与 Agent 3 的 `industry_position.revenue_geography_note` 交叉核对后写清假设）、**品牌与行业认可度**（奖项、头部客户背书、第三方排名等须有来源）。数据来自 Phase 1 财报与 **Agent 3**（`news_intel.json` → `industry_position`）的 web 检索合成；Phase 2 写入 **`financial_analysis.json` → `summary_para_4`**。

**投资逻辑一句话（结论前置）：** "公司凭借[核心优势]，有望在[时间窗口]内实现[具体预期结果]，主要驱动因素为[X]和[Y]。"

### 财务概览写作规范

**KPI 四卡（`{{KPI1_DIRECTION}}` … `{{KPI4_DIRECTION}}` 与对应 `{{KPI*_CHANGE}}`）：**

- **`{{KPI*_VALUE}}` 数字格式：** 亏损、负利润率、负 FCF 等须在 **KPI 主数值** 用 **ASCII 负号 `-` 与数字连写**，例如 `-22.3%`、`-16.4亿美元`、`-1.2亿美元`。**KPI 主数值不要加「约」**——四舍五入后的**直接数字 + 单位**即可；**禁止**用「净亏损约」「约负」「负向约」等代替负号本身。正文叙述如需强调口径不确定，可另用「约」。（读者已对红底有预期，主数字应一眼可读、可与表格核对。）
- 卡片与 `.kpi-change` 的 class **须一致**，取值仅为 **`up` / `down` / `neutral-kpi`**（与 `agents/report_validator.md` 一致）。
- **营业收入、归母净利润、净利率**：通常「越高越好」用 `up`（绿），走弱用 `down`（红）。
- **自由现金流（第三卡）易错点：** 若 **当年与上年 FCF 均为负**，但当年负得更少（向零收窄），**不得**使用 `up`（绿）+「明显改善」「显著改善」等无数字措辞——读者会误以为现金流已健康。应：
  - 使用 **`neutral-kpi`**（**红色左边框 + 浅红底**，与归母亏损等亏损卡视觉一致；**仅**语义上区别于 `up`，避免绿色），且
  - `{{KPI3_CHANGE}}` **必须写出可核对数字**，例如「同比亏损收窄约 X 亿美元（FCF 仍未转正）」或等价表述（金额可按财报百万美元换算为「亿美元」口径，与 KPI 主数值一致）。
- 若 FCF 已转正在零以上且高于上年，再用 `up`。
- 与本节「避免无法量化的『显著』」一致：**同比变化句应带具体幅度**（金额、百分点或bps），除非披露确实无法量化并在 `financial_analysis.json` → `notes[]` 说明。

**指标明细表最后一列（`同比变动`）：**

- 这列是阅读当年/上年数据后的**结论性定调**，不是计算过程展示。**不得**填写 `+0.62%`、`-1.4pct`、`+0.00%` 这类裸数字。
- 默认使用 `references/financial_metrics.md` 的受控词表：`显著改善`、`改善`、`基本持平`、`恶化`、`显著恶化`、`权益缺口收窄`、`权益缺口扩大`、`期末股东权益为负`、`不适用`。
- 若需要展示精确幅度，把数字写入当年/上年值、趋势卡正文或脚注；不要挤进 `同比变动` 单元格。

趋势分析写法（以归母净利润为例）：

**同比增长时：**
"FY2025年归母净利润同比增长X.X%至XX亿美元，主要受益于[驱动因素1]及[驱动因素2]，部分被[拖累因素]所抵消。"

**同比下滑时：**
"FY2025年归母净利润同比下滑X.X%至XX亿美元，主要受[原因1]及[原因2]拖累，我们认为[是否是一次性因素/是否预计改善]。"

**最新经营更新（第二节第四张趋势卡，位于自由现金流与地区收入之间）：**

- **数据归属：** 最近一期 **10-Q / 季报 / 半年报** 的抓取与结构化由 **Phase 1 → `agents/financial_data_collector.md`（Agent 1）** 写入 **`financial_data.json` → `latest_interim`**；本卡**不得**在缺失 `latest_interim` 时虚构季报数字。  
- 数据源：**最近已披露**的 10-Q、TTM、或中期业绩；与上方三张趋势卡的**完整财年同比**区分开。  
- **同比 vs 环比：** 正文**优先写同比（YoY）**——「最近一季」对**上年同季**，或「财年至今 YTD」对**上年同期 YTD**（与公司财年对齐），以削弱季节性误导。若披露或投资逻辑强调**相对上一季**的变化，可**补充一句环比（QoQ）**，并标明「环比」；单季环比受季节性影响大，**不应单独替代**同比作为主增速表述。  
- **首句必须写清期间**，例如：「据 FY2026 第一季度 Form 10-Q（截至 …，申报日 …）」「FY2026 财年前三季度同比 …」。避免读者误以为仍是 FY2025 vs FY2024 全年。  
- 内容侧重**边际变化**：收入/毛利/经营现金流的短期动能、管理层指引、一次性项目提示；无可靠季报时如实说明「中期披露不足」。

### 波特五力分析写作规范

每个透视（公司层面 / 行业层面 / 前景展望）合计约 300 字，须覆盖全部五力。

**第五节 HTML 正文格式（Phase 5 强制，三 tab 一致）：**

- **`{{PORTER_COMPANY_TEXT}}`、`{{PORTER_INDUSTRY_TEXT}}`、`{{PORTER_FORWARD_TEXT}}`** 均使用 **无序列表**：`<ul style="margin:0;padding-left:1.25em;">` 内 **恰好 5 个 `<li>`**，顺序固定为：**供应商议价能力 → 买方议价能力 → 新进入者威胁 → 替代品威胁 → 行业竞争强度**（与雷达轴与右侧评分列表顺序一致）。
- **不要**在每条 `<li>` 内用「**力名称（X/5）**：」作**标题式起句**——分数与力的标签已在**雷达图**与**右侧彩色分值列表**中展示；正文以**论据与分析**为主。
- **合议与 QC 透明（强制）：** 双 QC 与 Analyst 合议后的**最终版**，每个 Porter `<li>` 都应把 QC 结论前置写清，但**前提是结论必须来自真实的 QC 流程**：`qc_porter_peer_a.json` / `qc_porter_peer_b.json` 提出挑战，`qc_resolution_merge.md` 裁定后，才决定该维度是**维持原评分**还是**从初稿分数调整到新分数**。**不得为了套格式虚构“初稿分数”或编造调分故事**。完整审计链与条目级 rationale 仍以 **`qc_audit_trail.json`** 及附录 / `qc_deliberation` 摘要为主。
- **合议句式（中文固定格式，Phase 5 强制）：** 每个力对应的 `<li>` 开头都应根据**真实合议结果**使用下列两种句式之一，且要**点名具体力名**（如供应商议价能力、买方议价能力），不要写成「本维度」这类代称：  
  **维持分数：**「**经QC合议，维持供应商议价能力为3分。……**」或「**经QC合议，决定将供应商议价能力评分维持3分不变。……**」  
  **调整分数：**「**经QC合议，决定将供应商议价能力评分从4分调整为3分。……**」  
  其中 **a、b 为 1–5 分制整数**；句号后继续写证据、争议点与投资含义。**只有当 `qc_audit_trail.json` / `porter_analysis.qc_deliberation` 能证明该维度确有 score change 时，才可使用“从 a 调整为 b”句式；否则一律写“维持 X 分”。** 若初稿误把成熟在位者写成新进入者、或误把竞争问题写成供应商问题，可在后文交代口径纠偏。**勿**用仅含「【QC修正：PA-xxx】」而无实质分析的工单体单独占段；合议编号可置于句末括注。
- 每条 `<li>` 内可用多句，用分号或逗号连接；禁止 Markdown（`**` 等）。
- 中间数据层 `porter_analysis.json` 中 `company_perspective` / `industry_perspective` / `forward_perspective` 各子段应为**不以「力名（X/5）：」标题起句**、但**以“经QC合议，维持/决定将…评分…”起句**的分析句，便于直接填入 `<li>`；相应子段的“维持/调整”动作必须与 `qc_audit_trail.json` 中的真实裁定一致，避免 JSON 与 HTML 两套口径，更不得出现 HTML 编造调分而审计轨迹没有记录的情况。

**具体性与命名（行业层面与公司层面均适用）：**
- **行业层面**描写产业结构（集中度、周期、资本强度）时，仍须写出**可核对的主体**：凡出现「在位者」「现有巨头」「主要竞争者」「头部厂商」等措辞，须在**同一句或紧邻一句**内列出该子行业**全球（或报告所选主市场）前 3–5 名**主要企业，或写「以 A、B、C 等寡头为主」一类表述；**禁止**仅用抽象集合名词带过。
- **新进入者威胁**若写成「来自现有厂商之间的节点竞赛 / 产能扩张」而非全新进入者，必须**点名**这些厂商是谁（与**行业内竞争**维度一致，避免读者不知道「巨头」指谁）。
- **公司层面**继续写本公司特有事实（合资、客户结构、与具体对手份额对比等）；与行业层面分工是「本公司细节 vs 赛道格局」，不是「行业层面不写公司名」。

**内容结构（与列表并行，不另写开篇重复分数的段落）：**
1. 每一力对应**一条 `<li>`**：2–3 句具体分析，尽量有数据或事实。
2. 关键风险或机遇可并入相应力的 `<li>`，不必单独第六段。

### 第三节：宏观因子传导阐释（`macro_factor_commentary` / `{{MACRO_FACTOR_COMMENTARY}}`）

- **由 `macro_factors.json` 生成**（见 `agents/macro_scanner.md` Step 7b），Phase 5 **原样粘贴** HTML，勿在模板里另写一套。
- **写法：** 机构研报口吻，说明「表中六项调整幅度之和」如何对应瀑布图「宏观调整」栏（合计）；**至少四类**因子要点名公司主业（如利率→资本成本与估值、GDP→终端需求、油价→能源/物流与风险偏好等），并与 **公司** 业务/地区结构挂钩，避免纯宏观教科书。

- **禁止 Markdown**；可用 `<p>…</p>`，与第二节趋势卡一致。

## 附录与页眉：「具体来源」列披露口径（必读）

读者关心的是**信息最初出自哪里**，而不是仓库里的中间文件名。**「具体来源」列写法定/权威发布方**；脚本与 JSON 可在工作底稿或 `notes[]` 里保留技术路径。

| 实际获取方式 | 附录「具体来源」建议写法 | 不要这样写 |
|--------------|--------------------------|------------|
| `scripts/sec_edgar_fetch.py` 拉取 `data.sec.gov` / submissions，写入 `sec_edgar_bundle.json` | **美国 SEC EDGAR**（可加括号：`data.sec.gov`，经本技能脚本拉取） | 仅写脚本名或 `sec_edgar_bundle.json` 作为唯一来源，让读者误以为是「非 SEC」 |
| 打开 **sec.gov Archives** 上的 Form **10-K / 10-Q** HTML/PDF（含 **MD&A、Note 16 Revenue** 等附注与讨论）摘录数字 | **美国 SEC EDGAR**（括号内写章节位置，如：`Form 10-K` MD&A、Note 16 Revenue） | 把「10-K MD&A」写成与 SEC 并列的第三方；MD&A 与附注**是申报文件组成部分**，来源仍是 **SEC** |
| 公司 **IR 网站**新闻稿 / 投资者关系 PDF（非 sec 托管） | **公司 IR**（可写域名） | 写 SEC |
| **Bloomberg** / **Reuters** / 财经媒体网页 | **Bloomberg**、**Reuters** 等媒体全名 | 写 SEC |
| Yahoo Finance、Macrotrends 等二级聚合 | **Yahoo Finance** 等（并标置信度中等为宜） | 写 SEC |

**原则一句话：**凡数字或表格**最终可追溯到向 SEC 提交或托管在 EDGAR 的法定披露**，附录统一标 **SEC**；**仅在非 SEC 渠道首次出现**时才标该渠道名。
